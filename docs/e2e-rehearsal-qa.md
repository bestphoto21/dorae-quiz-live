# End-to-End 리허설 QA 체크리스트

## 목적

실제 행사 전 운영자, 참가자, 스크린 환경을 기준으로 처음부터 끝까지
리허설을 진행하고 문제를 기록하기 위한 문서입니다.

## 1. 리허설 기본 원칙

- 실제 행사와 최대한 같은 환경에서 테스트합니다.
- 로컬보다 Production URL 또는 행사와 같은 배포 환경에서 테스트합니다.
- 운영자 PC, 스크린 PC, 참가자 휴대폰을 분리해서 테스트합니다.
- 새 기능 추가보다 기존 기능 안정성 확인을 우선합니다.
- 개인정보는 사용하지 않고 테스트용 가명만 사용합니다.
- 문제를 발견하면 즉시 버그 리포트 또는 현장 이슈 로그에 기록합니다.

## 2. 리허설 준비물

- 운영자 PC 1대
- 스크린 출력 PC 또는 별도 브라우저 1개
- 참가자 역할 휴대폰 2-3대
- 관리자 계정
- 테스트 행사 `event_code`
- 테스트 경품 2-3개
- 테스트 설문 1개 이상
- 테스트 Q&A 질문 2-3개
- 스피커 또는 오디오 출력 장치

## 3. 테스트 URL 기록란

아래 URL은 실제 `eventId`와 `eventCode`에 맞춰 바꿔 기록합니다.

| 항목 | URL | 확인 |
| --- | --- | --- |
| 관리자 통합 운영 콘솔 | `/admin/events/[eventId]/operations` | [ ] |
| 운영 체크리스트 | `/admin/events/[eventId]/checklist` | [ ] |
| 행사 설정 | `/admin/events/[eventId]/settings` | [ ] |
| 참가자 입장 | `/e/[eventCode]/join` | [ ] |
| 참가자 플레이 | `/e/[eventCode]/play` | [ ] |
| 참가자 설문 | `/e/[eventCode]/survey` | [ ] |
| 스크린 | `/screen/[eventCode]` | [ ] |
| 럭키드로우 | `/admin/events/[eventId]/draw` | [ ] |
| 결과 다운로드 | `/admin/events/[eventId]/exports` | [ ] |
| 리허설 초기화 | `/admin/events/[eventId]/rehearsal` | [ ] |

## 4. 리허설 전 사전 점검

- [ ] 최신 배포가 Vercel Ready 상태입니다.
- [ ] Supabase migration 004-008이 적용되어 있습니다.
- [ ] 관리자 로그인이 가능합니다.
- [ ] 참가자 화면 설정의 제목, 설명, 기능 ON/OFF가 행사 성격에 맞습니다.
- [ ] 스크린 화면 문구가 행사 성격에 맞습니다.
- [ ] 경품이 1개 이상 등록되어 있습니다.
- [ ] 설문에 질문이 1개 이상 있습니다.
- [ ] 스크린 PC에서 `/screen/[eventCode]`가 정상 표시됩니다.
- [ ] 참가자 휴대폰에서 QR 접속이 가능합니다.
- [ ] 효과음을 사용할 경우 스크린에서 효과음 켜기를 눌렀습니다.

## 5. 리허설 시나리오 A: 기본 입장 흐름

순서:

1. 통합 운영 콘솔에서 QR 입장 안내를 송출합니다.
2. 참가자 휴대폰으로 QR에 접속합니다.
3. 참가자 이름과 소속을 테스트용 가명으로 입력합니다.
4. 참가자 play 화면에 진입합니다.
5. 관리자 화면에서 참가자 수 증가를 확인합니다.
6. 운영 체크리스트에서 참가자 수 반영을 확인합니다.

확인 기준:

- [ ] QR이 스크린에서 잘 보입니다.
- [ ] 모바일 접속이 빠르게 열립니다.
- [ ] 참가자 등록 완료 안내가 명확합니다.
- [ ] 중복 입장 시 기존 참가자 세션 또는 안내가 자연스럽게 동작합니다.

## 6. 리허설 시나리오 B: 1분 설문 흐름

순서:

1. 통합 운영 콘솔에서 1분 설문을 시작합니다.
2. 스크린이 `survey_active` 화면으로 전환되는지 확인합니다.
3. 참가자 play 화면에 진행 중 설문 카드가 표시되는지 확인합니다.
4. 참가자 휴대폰에서 설문을 작성합니다.
5. 선택지와 rating 선택 표시가 즉시 바뀌는지 확인합니다.
6. 제출 버튼 pending 상태를 확인합니다.
7. 제출 완료 안내를 확인합니다.
8. 관리자 화면에서 제출 수 증가를 확인합니다.
9. 스크린에서 제출률 증가를 확인합니다.
10. 1분 후 자동 마감 또는 lazy close 동작을 확인합니다.
11. 마감 후 제출이 차단되는지 확인합니다.

확인 기준:

- [ ] 남은 시간 표시가 정상입니다.
- [ ] 제출률 갱신이 정상입니다.
- [ ] 제출 후 수정 불가 안내가 표시됩니다.
- [ ] 마감 후 제출이 차단됩니다.
- [ ] 화면 깜빡임이 심하지 않습니다.

## 7. 리허설 시나리오 C: 설문 제출자 대상 럭키드로우

순서:

1. 럭키드로우 관리 페이지에 접속합니다.
2. 추첨 대상을 설문 제출자로 선택합니다.
3. 방금 진행한 설문을 선택합니다.
4. 후보자 수를 확인합니다.
5. 경품을 선택합니다.
6. 새 당첨자 추첨을 실행합니다.
7. 스크린에서 롤링 후 당첨 발표가 표시되는지 확인합니다.
8. 콘페티가 표시되는지 확인합니다.
9. 효과음 ON 상태라면 효과음이 들리는지 확인합니다.
10. 당첨자 CSV에 기록되는지 확인합니다.

확인 기준:

- [ ] 설문 제출자만 후보가 됩니다.
- [ ] 새 추첨과 최근 결과 재송출이 구분됩니다.
- [ ] 당첨 발표가 같은 `/screen/[eventCode]`에서 표시됩니다.
- [ ] 콘페티와 효과음이 정상입니다.
- [ ] `participant_id`, phone, email이 스크린과 CSV에 노출되지 않습니다.

## 8. 리허설 시나리오 D: Q&A 흐름

순서:

1. 참가자 화면 설정에서 Q&A 기능이 ON인지 확인합니다.
2. 참가자 화면에서 테스트 질문을 제출합니다.
3. 관리자 Q&A 페이지에서 질문을 확인합니다.
4. 질문을 승인하고 송출합니다.
5. 스크린이 Q&A 화면으로 전환되는지 확인합니다.
6. Q&A 해제 또는 대기 화면 송출을 실행합니다.

확인 기준:

- [ ] Q&A OFF일 때 참가자 화면에서 Q&A가 숨겨집니다.
- [ ] Q&A ON일 때 질문 제출이 가능합니다.
- [ ] 스크린에 개인정보 없이 질문이 표시됩니다.
- [ ] 마지막 송출 버튼 기준으로 화면이 전환됩니다.

## 9. 리허설 시나리오 E: 빠른 화면 전환

통합 운영 콘솔에서 아래 순서로 진행합니다.

1. 대기 화면 송출
2. QR 입장 안내 송출
3. 휴식 화면 송출
4. 설문 제출 현황 송출
5. Q&A 송출
6. 럭키드로우 준비 화면 송출
7. 최근 당첨 결과 재송출
8. 대기 화면 송출

확인 기준:

- [ ] 같은 `/screen/[eventCode]` 하나에서 전환됩니다.
- [ ] 마지막 송출 버튼이 우선합니다.
- [ ] 화면이 새로고침처럼 크게 깜빡이지 않습니다.
- [ ] 이전 화면 상태가 남지 않습니다.

## 10. 리허설 시나리오 F: 결과 다운로드

다운로드 항목:

- 참가자 명단 CSV
- 설문 응답 CSV
- 설문 제출자 CSV
- 당첨자 CSV
- Q&A CSV
- 운영 로그 CSV

확인 기준:

- [ ] 파일 다운로드가 가능합니다.
- [ ] Excel에서 한글이 깨지지 않습니다.
- [ ] 빈 데이터여도 헤더가 있습니다.
- [ ] `participant_id`, phone, email, raw `screen_payload`가 없습니다.

## 11. 리허설 시나리오 G: 리허설 데이터 초기화

순서:

1. 리허설 초기화 페이지에 접속합니다.
2. 현재 데이터 수를 확인합니다.
3. 필요한 항목만 체크합니다.
4. `RESET [event_code]`를 정확히 입력합니다.
5. 브라우저 confirm을 확인합니다.
6. 초기화를 실행합니다.
7. 스크린이 waiting 상태인지 확인합니다.
8. 참가자, 응답, 당첨자, Q&A가 선택한 범위대로 초기화됐는지 확인합니다.
9. 설문지, 문제, 경품, 설정은 남아 있는지 확인합니다.

확인 기준:

- [ ] 설정과 콘텐츠가 보존됩니다.
- [ ] 선택한 운영 데이터만 삭제됩니다.
- [ ] `operation_logs`에 초기화 기록이 남습니다.
- [ ] 실제 행사 데이터에서는 신중히 사용해야 함을 운영자가 이해했습니다.

## 12. 최종 합격 기준

- [ ] 운영자 화면에서 모든 핵심 버튼 위치가 명확합니다.
- [ ] 참가자 휴대폰에서 막히는 구간이 없습니다.
- [ ] 스크린 전환이 안정적입니다.
- [ ] 설문 제출률이 정상입니다.
- [ ] 럭키드로우가 정상입니다.
- [ ] CSV 다운로드가 정상입니다.
- [ ] 리허설 초기화가 정상입니다.
- [ ] 개인정보 노출이 없습니다.
